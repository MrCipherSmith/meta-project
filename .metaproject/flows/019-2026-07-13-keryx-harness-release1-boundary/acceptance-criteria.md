# Acceptance Criteria

Rules:

- Criteria lines use the exact format `- ACn: <criterion>`.
- After `flow freeze` this file is checksum-protected: any edit outside
  `keryx flow ac update` fails every gate and status transition.
- Completion requires every ACn to be confirmed via
  `keryx flow ac confirm <id> <ACn>`.

## Criteria

- AC1: H-01 provider negatives (offline, non-vacuous) — a consolidated red-team suite under `src/harness/provider/anthropic/` drives the W14 `AnthropicProvider` (recorded fixtures + mocked `fetch`, NO live network) and asserts fail-closed handling of the deferred provider negative families — timeout / stalled body (a deadline `AbortSignal` firing during the body read yields a terminal `cancelled` `provider_error`, never an uncaught throw), rate-limit (429 with/without `retryAfterMs`), malformed event, truncation (torn SSE with no `model_end`), zero-byte / empty response body (→ a fail-closed `malformed` `provider_error`, never a silent zero-event success), egress-deny (private/loopback/metadata incl. encoded base URL), cancellation (AbortSignal → `cancelled`), authentication (401) — each with the correct `ProviderErrorKind` + `retryable` and no credential leak. Two fail-closed gaps the suite surfaced in the W14 adapter (the stalled-body throw and the empty-body silent success) are CLOSED by a minimal additive fix to `anthropic-provider.ts` (a guarded body read + an empty-body terminal error), and the corresponding tests assert the fail-closed behavior. The suite is non-vacuous (it drives `stream()` to a terminal event) and deterministic (no `Date.now`/`Math.random`).
- AC2: E-01 Release 1 evidence matrix — `docs/decisions/keryx-harness/E-01-release1-evidence-matrix.md` maps each Release 1 capability (W8 resume, W9 branch/compaction, W10 mutation, W11 flow integration, W12 child agents, W13 parallel scheduling, W15 hardening incl. SSRF/NaN/extension/budget, W14 real provider) to a real source file, a real test, and a real commit hash (from c279e3a / 33f8e8d / 8ed5373 / d2f8ca4 / 550f372 / 8ec1016 / de46260 / 109c63c), and every claim is marked implemented / planned / deferred (traceability gate); a research-ledger update and migration notes are included. The doc edits NO runtime or frozen file.
- AC3: E-02 Release 1 review package — `docs/decisions/keryx-harness/E-02-release1-review-package.md` is an independent multi-lens review (architecture / contract / logic / security / testing-replay / performance / Gherkin over S-01…S-12) of the built Release 1, recording findings by severity and a GO / NO-GO verdict with any P0/P1; the review is read-only (the source under review is NOT modified — a normalized managed review package only).
- AC4: E-03 Release 1 handoff (gated) — IF E-02's verdict is GO with no BLOCKER/P0/P1, `docs/decisions/keryx-harness/E-03-release1-handoff.md` promotes the roadmap/package and provides the Release 1 → Release 2 handoff (DAG of what is built, a frozen-AC proposal for Release 2, gates, constraints, and out-of-scope), explicitly listing the deferred `@release-2` scenarios as the next track; IF E-02 is not clean, the blockers are recorded instead and reported. The doc edits NO runtime or frozen file.
- AC5: No regression / scope / determinism / deferrals — `tsc --noEmit` is clean and the full `bun test` suite is ≥ the pre-change baseline of 1150 pass with the new H-01 tests green and 0 fail, entirely OFFLINE (no live network); the ONLY runtime-code change is a minimal ADDITIVE fail-closed fix to `src/harness/provider/anthropic/anthropic-provider.ts` closing the two gaps the H-01 suite surfaced (guarded body read → `cancelled`; empty body → `malformed`) — no existing allow-path or prior test is altered; E-01/E-02/E-03 change no runtime/production code; new docs live under `docs/decisions/keryx-harness/` (Release 1 files; the R0 boundary docs are NOT overwritten); behavior is deterministic; no new production dependency (`dependencies` `{}`), no provider SDK, no network in tests; the harness never writes flow.json (D-02); the frozen requirements package, canonical contract schemas, `src/eval/`, `src/contracts/`, and ADR-0001…0004 are NOT modified. Release 2 (`@release-2` scenarios) is documented as the next track, NOT implemented here.

# Acceptance Criteria — flow 060 (OpenTUI shell Phase 1)

- AC1: An ADR (`docs/decisions/keryx-harness/ADR-0005-opentui-shell-dependency.md`) ratifies adding `@opentui/core` as keryx's first UI-facing OPTIONAL native dependency: status, decision, rationale, the optionalDependencies + dynamic-import-only + graceful-fallback contract, portability/platform-binary implications, and the zero-`dependencies`-floor invariant it does NOT change.
- AC2: `@opentui/core` is added to `package.json` `optionalDependencies` (NOT `dependencies`); the AC15 pin (`src/testing/block-d-no-network.test.ts`) is updated to the new exact set with a rationale comment; `dependencies` stays `{}`; the no-top-level-import guard still passes (the package is imported ONLY via dynamic `import()`).
- AC3: `src/tui/tui-shell.ts` provides a `TuiShell` renderer skeleton that implements the `AgentIO` hook surface by appending PLAIN-TEXT blocks to an OpenTUI transcript (write/onAssistantText/onReasoning/onUsage/onToolCall/onToolResult/onSystem), plus `launchTuiAgentShell(deps)` that drives `runAgentTurn` from a `split-footer` composer. `--tui` opts in from `shellCommand` (agent mode); on no-TTY / absent optional dep / init failure it returns false and the readline shell runs (fallback verified).
- AC4: A headless test drives a SCRIPTED provider through `runAgentTurn` with the `TuiShell` IO against `@opentui/core/testing`'s `createTestRenderer`, and asserts the captured frame contains the streamed assistant text and a tool line — proving the driver→TUI render path without a real TTY. `bunx tsc --noEmit` clean; `bun test` green with no reduction from the pre-dep baseline (1494 non-TUI + new TUI tests); `runAgentTurn`, the pure helpers, chat mode, and `roleLabel` are unchanged. Markdown/gutter/collapse parity is explicitly deferred to Phase 2.

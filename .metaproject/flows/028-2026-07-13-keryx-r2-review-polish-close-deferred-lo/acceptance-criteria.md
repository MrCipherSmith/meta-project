# Acceptance Criteria

Rules:

- Criteria lines use the exact format `- ACn: <criterion>`.
- After `flow freeze` this file is checksum-protected: any edit outside
  `keryx flow ac update` fails every gate and status transition.
- Completion requires every ACn to be confirmed via
  `keryx flow ac confirm <id> <ACn>`.

## Criteria

- AC1: Tool-call aggregate accounting fail-closed (H) — `inheritBudget` (`src/harness/child/isolation.ts`) DENIES a child that omits `maxToolCalls` when the parent HAS a `maxToolCalls` budget (fail-closed, symmetric with the existing rule that denies a child requesting a cap when the parent has none); the doc comment is updated to match. A parent WITHOUT a `maxToolCalls` budget still grants a cap-less child on runtime alone (unchanged), and a child requesting a cap ≤ parent remaining is still granted (unchanged). The W13 `planWaves` scheduler propagates this deny (a wave with a cap-less child under a capped parent is denied). No fail-open path remains where N cap-less children collectively bypass a finite parent tool-call budget. (If this required a real refactor of W12/W13, it was reported instead — not forced.)
- AC2: Contained-process evidence is joinable (A) — `RunContainedProcessInput` carries (optional) run/session/correlation ids and `runContainedProcess`'s `buildEvidence` stamps `causal.runId`/`sessionId`/`correlationId` from them (rather than freshly minted `idSeq` values), so the contained-process evidence joins back to its originating run; when the ids are absent the prior `idSeq` fallback is preserved (existing callers/tests unaffected).
- AC3: Single provider factory + DRY (B, C) — a single `makeProvider(...)` factory (under `src/harness/provider/`) is the sole place that instantiates `AnthropicProvider`/`OllamaProvider`/`FakeProvider` (incl. the missing-`ANTHROPIC_API_KEY` fallback), and both `src/commands/shell.ts` and `src/commands/harness.ts` route through it; the duplicated `USAGE` string (harness.ts), selection-apply block (shell.ts), and divergent menu-loop idiom (select.ts) are de-duplicated — all behavior-preserving (existing tests stay green).
- AC4: Evidence-quality corrections (D, E, F, G) — (D) `provenance.test.ts` now exercises the default-root fallback (omitting `registrationProvenance` → a `trustLevel:"derived"` record taint-linked to the registry root); (E) `bound-wave.ts` planning-time per-attempt evidence no longer carries a `child-result:DONE` disposition that reads as completion (a neutral/PLANNED disposition is used) while per-attempt isolation is intact — and NO frozen prior-wave disposition enum was extended (if that was required, E was reported/deferred instead); (F) the `runContainedProcess` outcome `evidenceRefs` use the SAME prefixed encoding as the receipt; (G) the real-adapter smoke guard is an observable assertion (not a tautology) and the suite is still CI-inert (0 real spawns without `KERYX_ALLOW_REAL_SUBPROCESS`).
- AC5: No regression / scope / invariants — `tsc --noEmit` is clean and the full `bun test` suite is ≥ the baseline (1325 pass / 2 skip) with the new tests green and 0 fail; behavior is deterministic (injected id/clock; no `Date.now`/`Math.random` in the offline cores); no new production dependency (`dependencies` `{}`), no real spawn in the offline suite; the executor/adapter/extension/scheduler write no flow.json (D-02); secrets (env values / API key) are never logged; edits to prior-wave modules are limited to the minimal fail-closed/traceability/disposition tightening (H, A, E) — no rewrite; the frozen requirements package, canonical schemas, `src/eval/`, `src/contracts/`, and ADR-0001…0004 are NOT modified.

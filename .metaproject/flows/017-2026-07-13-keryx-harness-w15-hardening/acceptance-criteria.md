# Acceptance Criteria

Rules:

- Criteria lines use the exact format `- ACn: <criterion>`.
- After `flow freeze` this file is checksum-protected: any edit outside
  `keryx flow ac update` fails every gate and status transition.
- Completion requires every ACn to be confirmed via
  `keryx flow ac confirm <id> <ACn>`.

## Criteria

- AC1: SSRF / private-egress hardening (additive) — `src/harness/mutation/guard.ts` denies a private/loopback/link-local/metadata egress destination expressed in ANY of these forms, in addition to the existing list: IPv6 loopback (`::1`, `[::1]`), decimal-encoded IP (`2130706433`), hex (`0x7f000001`), octal (`0177.0.0.1`), short form (`127.1`), the full `172.16`–`172.31` range, `0.0.0.0`, CGNAT `100.64.`, and case-insensitive `localhost`/`LOCALHOST`. The broadening is ADDITIVE (it only ever DENIES more; every pre-existing allow-path and deny-path of `guardAction` is preserved and its prior tests still pass).
- AC2: NaN-date fail-closed — `src/harness/mutation/approval.ts` treats an unparseable or NaN `expiresAt` or `now` as INVALID and fails closed (the approval is denied / expired), never fail-open as "unexpired"; the change is additive and backward-compatible (valid timestamps behave exactly as before; all prior approval tests still pass).
- AC3: SC_R18 unregistered extension denied + H-02 contract — `src/harness/extension/registry.ts` provides a PURE, fail-closed `registerExtension` that rejects an extension lacking a pinned manifest OR a capability grant, producing no discovery-time mutation or authority (`SC_R18_UNREGISTERED_EXTENSION_DENIED`); `docs/decisions/keryx-harness/H-02-deferred-extension-capability-contract.md` defines the deferred extension capability grants + isolation model WITHOUT enabling them and cites the `@release-2` extension scenarios (`SC_R18_REGISTERED_EXTENSION_PROVENANCE`, `SC_R08_EXTENSION_ESCALATION_REQUIRES_POLICY`) as explicitly later scope. Frozen ADR-0001…0004 are NOT modified.
- AC4: SC_R16 budget reconciliation + recovery/replay/migration regression-lock — `src/harness/budget/reconcile.ts` provides a PURE, deterministic reconciliation over planned/reserved/consumed/remaining with a reliability tag (`exact|estimated|unknown`) that fails closed on over-consumption (consumed > reserved) or negative remaining (`SC_R16_BUDGET_RESERVATION`); and the existing fail-closed / recovery invariants are regression-locked by non-vacuous tests: W12 `inheritPolicy`/`inheritBudget`, W13 scheduler ceilings, and W10 guard/approval hold under adversarial red-team inputs; W8 resume survives crash/torn-write cut points with outcome-unknown blocking unsafe retry; W7 replay is effect-free; read-time `schemaVersion` migration is deterministic; performance/SLO bounds are measured deterministically (bounded event/context growth, no unbounded accumulation).
- AC5: No regression / reuse / scope / determinism / deferrals — `tsc --noEmit` is clean and the full `bun test` suite is ≥ the pre-change baseline of 1008 pass with the new tests green and 0 fail; every hardening change to an existing module is ADDITIVE (no allow-path altered, no prior test flipped); new runtime code lives under `src/harness/{extension,budget}/` only; behavior is deterministic (injected clock/id, no `Date.now`/`Math.random`); no new production dependency (`dependencies` `{}`), no provider SDK, no network, no real fs mutation in tests; the harness/child/scheduler/extension/budget helpers never write flow.json (D-02); the frozen requirements package, canonical contract schemas, `src/eval/`, `src/contracts/`, and ADR-0001…0004 are NOT modified. Explicitly DEFERRED (not gated here): the RP-01 / real-provider negative families of H-01 (→ post-W14 H-01 re-run) and the `@release-2` extension scenarios (documented in H-02 as later scope).

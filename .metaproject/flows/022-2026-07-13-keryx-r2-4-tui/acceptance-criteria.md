# Acceptance Criteria

Rules:

- Criteria lines use the exact format `- ACn: <criterion>`.
- After `flow freeze` this file is checksum-protected: any edit outside
  `keryx flow ac update` fails every gate and status transition.
- Completion requires every ACn to be confirmed via
  `keryx flow ac confirm <id> <ACn>`.

## Criteria

- AC1: Provider/model detection ‚Äî `src/commands/select.ts` `detectProviders(deps)` (over an injected `fetch` + `env`) returns the available providers with their models: `ollama` lists its CHAT models from `{baseUrl}/api/tags` (embedding families excluded) and is fail-SOFT when the server is unreachable / non-200 (reported "not available", never throws); `anthropic` is available only when `ANTHROPIC_API_KEY` is set in the injected env (with a static known-model list); `fake` is always available. Deterministic (no `Date.now`/`Math.random`).
- AC2: Interactive picker (no hardcoded default) ‚Äî `pickProviderModel(io, detected)` renders a numbered provider menu then a numbered model menu to `io.write` and returns the `{provider, model, baseUrl?}` chosen from the injected `io.lines`; an out-of-range / non-numeric choice is handled without a crash (re-prompt or a deterministic safe fallback). `shellCommand` runs `detectProviders` + `pickProviderModel` when no `--provider` is given (the previous hardcoded `provider="ollama"`/`model="llama3.1:latest"` default is removed), and skips the picker when `--provider X [--model Y]` is supplied.
- AC3: Slash commands + credential safety ‚Äî the `runShell` core (flow-021, reused) additionally handles `/models` (list the current provider's models and switch on a numeric pick), `/provider` (re-run selection), and `/connect` (write guidance on setting `ANTHROPIC_API_KEY`); the `ANTHROPIC_API_KEY` credential is NEVER stored, entered, or logged ‚Äî it is only read from env, and `/connect` merely explains how the user sets it. The existing `/help`, `/clear`, `/model`, `/exit`, `/quit` still work.
- AC4: Adapter-over-ports / reuse / egress (SC_R13_TUI intent) ‚Äî R2-4 changes NO runtime contract: it is an adapter over the stable CLI/JSONL-RPC ports, reusing the flow-021 `runShell` core + the W14/W20 provider adapters + the W15 SSRF guard + the W20 loopback egress opt-in UNCHANGED (composition only; the only edits to prior modules are additive `shell.ts` + a `cli.ts` help note; the new code is `src/commands/select.ts`). The ollama `/api/tags` probe uses the same configured (default loopback) base URL under the reused egress posture; chat egress stays loopback-gated. The shell/CLI never write flow.json (D-02).
- AC5: No regression / offline / deps / scope ‚Äî `tsc --noEmit` is clean and the full `bun test` suite is ‚â• the pre-change baseline of 1210 pass with the new tests green and 0 fail, and the ENTIRE automated suite runs OFFLINE (injected IO/fetch/env + FakeProvider; no real TTY, no live network ‚Äî the Ollama path is exercised only in the manual live smoke); behavior in the tested core is deterministic (no `Date.now`/`Math.random`); no new production dependency ‚Äî `dependencies` REMAINS `{}` (`node:readline` stdlib only; NO TUI framework/SDK); new code lives in `src/commands/select.ts` with additive-only edits to `src/commands/shell.ts` and `src/cli.ts`; the frozen requirements package, canonical contract schemas, `src/eval/`, `src/contracts/`, and ADR-0001‚Ä¶0004 are NOT modified. R2-1/R2-2/R2-3/R2-5 are out of scope. A live smoke of bare `keryx` against the local Ollama detects providers, lets the user pick provider+model, and streams a real multi-turn reply (recorded in the flow journal; not a CI gate).

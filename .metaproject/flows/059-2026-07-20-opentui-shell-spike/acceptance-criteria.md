# Acceptance Criteria — flow 059 (OpenTUI Phase 0 spike)

- AC1: (R1 native-install GATE) `bun add @opentui/core` resolves and installs on darwin-arm64 without a Zig toolchain, and the native module loads at runtime (an import/`createCliRenderer` smoke resolves the native binding, or the failure mode is captured). Pass/fail + evidence recorded in the spike report.
- AC2: (R3 API map) The concrete `@opentui/core` primitives for a text input, a select/list dropdown, a scrollable region, plus keyboard/focus + resize entry points are identified from the installed package exports/types and written into the spike report, mapping the specification's `(SPIKE)` items to real names.
- AC3: (proof) A minimal opt-in `keryx shell --tui` prototype exists (isolated; the readline shell stays default and untouched) rendering a static transcript + an input with a live `/` dropdown over 3-4 dummy commands. `bunx tsc --noEmit` is clean, and on no-TTY/unsupported/init-failure it does not crash (falls back or no-ops). Real-terminal look/scrollback validation is handed to the user.
- AC4: (report + gate) A spike report `report.md` records R1-R5 findings (native install, inline-vs-fullscreen + scrollback, component API, license, cold-start) and a go/adjust/fallback recommendation. `bun test` stays green (no reduction from baseline 1494); no production path changes behaviour unless `--tui` is passed.

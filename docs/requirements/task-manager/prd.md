# Task Manager: PRD

Version: 0.1.0
Status: production-ready scope frozen (see [specification.md](specification.md) section 2 and [brainstorm.md](brainstorm.md) sections 2-5)

## 1. Purpose

Task Manager делает работу агентов управляемой: каждая стори проходит как flow -
от инициализации (контекст, формализация, план, замороженные критерии приёмки)
через реализацию (имплементатор/оркестратор) до завершения (гейты, PR, отчёт в
issue). Люди задают работу и принимают решения; агенты исполняют; CLI хранит
истину о состоянии и не даёт срезать углы.

## 2. Users

- **Разработчик/владелец**: даёт вход (описание или issue), отвечает на интервью, принимает результат.
- **Оркестратор-агент**: запускает flow-init, ведёт flow через flow-manager, вызывает completion.
- **Имплементатор-агент**: исполняет план по задачам, создаёт draft PR; не может менять AC и статусы.
- **Completion-агент**: перепроверяет, гоняет гейты, комментирует issue.

## 3. Problem

Агентная работа без системы рассыпается: контекст собирается заново каждый раз,
критерии приёмки «плывут» по ходу реализации, статус живёт в голове агента,
завершение не проверяется. Нужна точка истины (state machine + файлы) и
механические гейты, которые агент не может обойти.

## 4. Goals

- Один вход: `flow init` из описания или GitHub issue.
- Автосбор детерминированного контекста из модулей Metaproject.
- Flow-пакет: описание, контекст, план, задачи, критерии приёмки.
- Замороженные AC (checksum) - правки только через task-manager CLI.
- Строгая state machine статусов; только flow-manager двигает статус.
- Draft PR как терминальное состояние имплементатора.
- Completion-гейты: AC + PR checks + health gate; провал возвращает в работу.
- Итог в issue-комментарии; без issue - предложение создать тикет.

## 5. Non-goals (v1)

- Запуск/мониторинг агентов самим модулем (это оркестратор/gdskills).
- Notion/Jira адаптеры (контракт заложен, реализация позже).
- UI/веб-доска; кросс-repo flows; параллельные flows на одну ветку.
- Автоматический merge PR (терминал - draft PR + done; merge - человек).

## 6. User stories

### 6.1 Инициализация из issue
Как оркестратор, я вызываю `flow init --issue <url>`, получаю папку flow с
контекстом (issue body, memory, gdgraph, health) и шаблонами плана/AC - чтобы
скил flow-init достроил формализацию, брейншторм и интервью.

Приёмка: папка `<NNN>-<date>-<slug>` создана; context.md содержит issue и
доступные артефакты; flow.json в статусе `initializing`.

### 6.2 Заморозка и старт
Как flow-init скил, после заполнения AC я вызываю `flow freeze` (checksum
зафиксирован, статус `ready`), затем `flow start` - имплементатор получает план.

Приёмка: freeze падает при пустых AC; после freeze любое изменение
acceptance-criteria.md мимо CLI валит гейты.

### 6.3 Управление ходом
Как flow-manager, я отмечаю задачи (`flow task done T2`), добавляю новые
(`flow task add`), и только я перевожу flow в `implemented` (`flow implemented
--pr <url>`), убедившись что draft PR создан.

Приёмка: смена статуса мимо валидных переходов - ошибка; implemented без PR - ошибка.

### 6.4 Завершение
Как completion-агент, я вызываю `flow complete`: гейты (AC подтверждены, PR
зелёный, health pass) проходят → `done` + краткий комментарий в issue; не
проходят → flow возвращается в `in-progress` с fix-нотами, и я перезапускаю
имплементатора.

Приёмка: комментарий в issue краток и по делу; без issue CLI подсказывает
спросить пользователя о создании тикета.

## 7. Success metrics

- Новый flow инициализируется одной командой из описания или issue.
- AC невозможно изменить незаметно: любой гейт ловит расхождение checksum.
- Статусная модель исключает «done без PR» и «done с красными checks».
- Оркестратор/имплементатор работают только через CLI и скилы - без прямых правок flow.json.

## 8. Risks

- gh недоступен/не авторизован → graceful degradation: локальный режим с warn, PR-гейт помечается skipped.
- Агенты правят flow.json руками → `flow check` валидирует целостность; учить скилами.
- Слишком жёсткие гейты для мелких задач → лёгкий путь: минимальный AC (1 критерий) допустим.
